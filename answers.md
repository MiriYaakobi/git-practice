תשובות לשאלות הבנה בשלבים:

שלב 0:

•	שלושת הרמות הן local, global ו system. תמיד הרמה הנמוכה גוברת על הרמה הגבוהה יותר.

•	כי אחרת ה-commits לא יקושרו לפרופיל.

•	ה core.autocrlf מונע התנגשויות של ירידות שורה (CRLF לעומת LF), ו input עדיף כי הוא מבטיח אחידות LF לכולם. 

•	היתרון ב-.gitattributes הוא אכיפה אוטומטית על כל מי שמשכפל את הפרויקט.

שלב 1:

•	נשתמש ב -h כשאנחנו כבר מכירים את הפקודה ורק רוצים להיזכר באופציות, ובנוסף הוא מדפיס ישירות לטרמינל שזה גם נח. נשתמש ב –help כאשר נרצה לדעת יותר לעומק או כשלא מכירים מספיק טוב, והוא נפתח בדפדפן או עמוד נפרד.

•	החלקים החובה הם אלו שלא נמצאים בתוך הסוגריים, ואלו שכן הם אופציונליים. שלוש הנקודות אומרות שאפשר לצרף עוד <name> בנוסף לבודד.

•	ה tldr מראה רק פקודות מוכרות ויומיומיות, וגם דוגמאות קוד מוכנות להעתקה, בעוד ש man page מתאר ממש את מבנה הפקודה ומפרט.

שלב 2:

•	כל שינוי בקובץ יגרום לשינוי מוחלט של ה hash ולכן הקומיטים לא דומים אחד לשני. זה חשוב כי מונע זיופים.

•	ה HEAD הוא המיקום הנוכחי שלנו- איזור העבודה. ה branch הוא מצביע לקומיט ספיציפי.

•	רק הגרסה שעברה add תכלל בקומיט. ה staging area מאפשר לנו לאסוף רק את השינויים שרוצים לארוז לקומיט, בנפרד ממה שקורה בקוד.

שלב 3:

•	כי הפקודה הזו משווה מול ה staging area, ולכן לא יוחזר כלום כי אין הבדל. כדי לראות מה מוכן לקומיט: git diff --staged.

•	כשרוצים להוסיף לגיט רק חלק מהשינויים בקובץ. דוגמה: תיקון באג ובאותו קובץ גם עושים עוד פעולה לדוג ניסוי כלשהו. אז משתמשים ב--p כדי להכניס לקומיט רק את התיקון ולהשאיר את הניסוי בחוץ.

•	הוא משכתב את ההיסטוריה ומשנה את ה Hash, וזה יהרוס למי שכבר משך את הקוד. מצב בטוח הוא רק על קומיט מקומי שעדיין לא עשו לו push לגיטהאב.

•	ב -S כי הוא מחפש בתוך הקוד עצמו מתי המחרוזת  נוספה או הוסרה ולא סתם בהודעות של הקומיט.

שלב 4:

•	כי reset מוחק את הקומיט מההיסטוריה וזה יוצר התנגשויות למי שכבר משך את הקוד אליו. revert יוצר קומיט חדש שמבטל את הפעולה, וככה ההיסטוריה נשארת בטוחה ומסונכרנת לכולם.

•	ה --soft מבטל את הקומיט אבל משאיר את הקבצים מוכנים ב staging area ובworking directory. ו --hard מוחק את הקומיט וגם דורס ומוחק את כל השינויים מ staging area ומ working directory לגמרי.

•	כדי לוודא שגיט לא ימחק קבצים חשובים ששוכחים  לגבות, ורק אז מריצים -f שמוחק בפועל.

•	בעזרת restore. אסור להשתמש ב reset --hard כי הוא מוחק את כל השינויים שעשו בקובץ ומחזיר אותו למצבו המקורי, במקום רק להוציא אותו מאזור ההכנה.

שלב 5:

•	כי זה רק יצירת מצביע ולא שכפול של הקבצים, לכן צוותים עובדים עם ענף לכל פיצ'ר בלי לפחד מבעיות של ביצועים או התנגשויות.

•	המיזוג יהיה  fast forward במידה והענף הראשי לא התקדם מאז שיצרו את הענף ולכן גיט פשוט מזיז את המצביע קדימה. אחרת, אם 2 הענפים התקדמו במקביל, לא יהיה.

•	כי הוא שומר בהיסטוריה את העובדה שהיה ענף פיצ'ר נפרד, וזה מקל על קריאת ההיסטוריה ועל ביטול הפיצ'ר.

שלב 6:

•	החלק שלמעלה שייך לענף הנוכחי, קל לזכור כי כתוב HEAD. החלק למטה שייך לענף שמתמזג, וקל לזכור כי כתוב את שם הענף.

•	כשמסתבכים במיזוג או כשיש הרבה קונפליקטים ומעדיפים לבטל הכל ולהתחיל נקי מחדש.

•	כל אחד מהם בוחר רק את הגרסה של עצמו וזה מסוכן כי זה זורק את השינויים של האחר לפח. Ours זה בחירה של הגרסה שלי, ו theirs הגרסה של הענף השני.

שלב 7:

•	כי rebase משכתב את ההיסטוריה ומשנה את ה hashes. אם מישהו כבר משך את הענף, הוא ייתקע עם קומיטים לא קשורים ויהיו התנגשויות וקומיטים כפולים כשינסה להסתנכרן. בכללי, לא עושים rebase לענף פומבי.

•	שניהם מועכים את הקומיט לתוך ההוא שלפניו. squash נותן לערוך ולשלב את שתי הודעות הקומיט יחד, ו fixup זורק לפח את ההודעה של הקומיט הנוכחי ומשתמש רק בזו של הקודם.

•	עדיף  mergeבסיום פיתוח, כשממזגים את ענף הפיצ'ר ל  main. לדוג בשביל לראות בבירור מתי הפיצ'ר התחיל והסתיים בהיסטוריה.

עדיף rebase במהלך הפיתוח כשרוצים למשוך עדכונים מ main לענף הפרטי. לדוג להתעדכן בקוד החדש בלי ליצור קומיט מיזוג מיותר, כדי לשמור על היסטוריה ליניארית.

•	כי git pull רגיל יוצר קומיט מיזוג אוטומטי רק כדי להסתנכרן עם השרת. ו pull --rebase לוקח את הקומיטים המקומיים ושם אותם בקו ישר בקצה שמעודכן, ושומר על היסטוריה חלקה.

שלב 8:

•	תרחיש לדוג כשבאמצע פיתוח פיצ'ר (והקוד לא שלם או שבור) ופתאום מבקשים לעבור לענף אחר כדי לתקן באג בייצור. לא נשתמש בקומיט כי קומיט נועד לשמור יחידת עבודה הגיונית ועובדת בהיסטוריה. שמירה של קוד שבור רק כדי להחליף ענף הורסת את ההיסטוריה בקומיטים בלי משמעות.

•	pop לוקח את השינויים חזרה ומוחק אותם מיד מרשימת ה stash.

apply לוקח את השינויים אבל משאיר אותם שמורים ברשימה.

•	תג מוער לשחרור גרסאות רשמיות ללקוח. הוא שומר אובייקט מלא בגיט עם מטא דאטה (לדוג מי יצר את התג, מתי וכו).

תג רגיל הוא כמו סימנייה זמנית במהלך העבודה. מצביע פשוט לקומיט בלי מידע נוסף.

שלב 9:

•	כי הוא לוקח קומיט בודד ומעתיק אותו לענף הנוכחי, בלי לקחת יחד איתו את שאר הפיתוחים או ההיסטוריה של הענף המקורי.

•	מריצים git reflog שמציג את כל ההיסטוריה של ה HEAD, מוצאים את ה hash של הקומיט שנמחק ומשחזרים.

•	כי הוא עובד על אלגוריתם של חיפוש בינארי- בכל צעד הוא חותך את כמות הקומיטים שצריך לבדוק לחצי, במקום לעבור עליהם אחד אחד באופן ליניארי.

שלב 10:

•	כי fetch רק מורידה נתונים מהשרת בלי לגעת בקבצים ובקוד. זה מאפשר לבדוק את השינויים לפני השילוב, ומונע קונפליקטים אוטומטיים ש pull (שמבצע ישר מיזוג) יכול לגרום.

•	כי --force דורסת את השרת בצורה עיוורת. אבל --force-with-lease מאפשרת את הדריסה רק אם מישהו אחר לא דחף קוד חדש לענף מאז העדכון האחרון.

•	כשעובדים ב forking workflow. origin הוא הקישור ל fork האישי בגיטהאב, ו upstream הוא הקישור לפרויקט המקורי המרכזי, שדרכו אפשר לקחת עדכונים כדי להישאר מסונכרנים.

שלב 11:

•	Merge commit מעביר את כל הקומיטים ויוצר קומיט מיזוג.

Rebase מעביר את כל הקומיטים בשורה אחת ליניארית בלי קומיט מיזוג.

Squash מועך את כל הקומיטים של הענף לקומיט אחד ב main.

עדיף Squash כשהענף מלא בקומיטים של טיוטה ורוצים להכניס ל main רק קומיט אחד מסודר שמסכם את כל הפיצ'ר.

•	נועל את main ככה שאף אחד לא יכול לדחוף אליו קוד ישירות. הוא מכריח את הצוות להעביר כל שינוי דרך pull request, דורש אישור ממתכנת נוסף, ומוודא שכל הבדיקות האוטומטיות עברו לפני המיזוג.

•	אירועים שקורים בגיטהאב, כמו פתיחת pull\_request או push.

uses קורא לפעולה ציבורית שמישהו אחר כבר כתב וארז מראש. run פשוט מריץ פקודת טרמינל שכותבים בעצמנו.

•	פשוט מוסיפים מילת מפתח כמו fixes או  resolveעם מספר ה issueלתוך התיאור של ה  pull request ואז כשה PR יתמזג ה Issue ייסגר אוטומטית.

שלב 12:

•	GitHub flow טוב לצוותים שמשחררים קוד ברצף או לאפליקציות ווב שמתעדכנות ישירות בשרת. Git flow זה למערכות מורכבות יותר שדורשות ניהול גרסאות ותמיכה בגרסאות ישנות גם.

•	קצת שינויים מקטינים את הסיכוי להתנגשות של קוד עם חברי צוות שכתבו באותו הזמן. ובנוסף כשיש יחסית קצת שורות הבודק יותר מרוכז ומוצא באגים אמיתיים.           

כשה RR גדול קשה לעקוב כמו שצריך והבדיקה פחות טובה.

•	Rebase כשהענף פרטי ועוד לא עשו לו push או שמישהו עוד לא משך. זה משאיר את ההיסטוריית קומיטים האישית ישרה ונקייה בשביל ה PR.

Merge כשהענף משותף עם אחרים או כשחשוב לתעד בהיסטוריה את הרגע של הסנכרון.

שלב 13:

•	גיט ימשיך לעקוב אחריו. כדי להפסיק לעקוב בלי למחוק מהדיסק מריצים git rm –cached.

•	Gitignore אומר לגיט מאיזה קבצים להתעלם לגמרי. Gitattributes אומר לגיט איך להתייחס לקבצים שנמצאים במעקב (כמו טיפול בירידות שורה).

•	גיט לא מסוגל לעקוב אחרי תיקיות ריקות. .gitkeep הוא מוסכמה של קובץ ריק בתיקייה כדי שגיט ישמור אותה בפרוייקט.

•	הקובץ mailmap מאחד שמות או אימליים שונים של אותו משתמש בתור אדם אחד. ו git-blame-ignore-revs מסתיר קומיטים כמו של פירמוט אוטומטי מפקודת ה blame כדי שיראו מי באמת כתב את הלוגיקה.

•	git rm –cached מוחק מהמעקב של גיט ומשאיר במחשב. Git rm מוחק מהמעקב של גיט וגם במחשב. Rm -rf זו פקודת מחיקה של מערכת ההפעלה שמוחקת מהדיסק.

•	Local- משפיע רק על הפרוייקט. Global- משפיע על המשתמש במחשב. System- משפיע על כל המחשב. דוג- הגדרת שם או אימייל.

•	הוא פותר את הבעיה שהריפו מתמלא בקבצים בינאריים ענקיים בכך שהוא שומר אותם בשרת חיצוני ושומר בגיט רק מצביע אליהם. מי שלא התקין את התוסף יקבל רק את קובץ המצביע וישאר בלי הקובץ האמיתי.

שלב 14:

•	פרוייקט האב שומר רק מצביע שמכיל את כתובת האינטרנט ואת ה hash של הקומיט. הוא לא שומר את הקבצים והקוד של פרוייקט הבן עצמו.

•	כי פקודת clone רגילה מורידה רק את המצביע של פרוייקט האב ולא את תוכן הפרוייקט החיצוני. כדי לתקן מריצים את הפקודה git submodule update --init –recursive.

•	פרוייקט האב נועל את פרוייקט הבן לקומיט ספציפי ולא לענף בשביל יציבות. הבעיה שאם כותבים שם קוד בלי לפתוח ענף חדש הקומיטים יהיו באוויר וימחקו ברגע שהמצביע יתעדכן.

•	Submodule כשמייבאים קוד ממקור חיצוני שלא קיים במאגר חבילות סטנדרטי וחייבים לקבע אותו לגרסה ספציפית.

Package Manager כשמייבאים ספריות קוד. זה מונע התנגשויות ופשוט לניהול.

Monorepo כשמפתחים כמה רכיבים שיש בניהם תלות וחייבים שכולם יעבדו תמיד על הגרסה העדכנית ביותר במקביל.

שלב 15:

a.	רמה בסיסית:

•	Git זו התוכנה עצמה שרצה מקומית על המחשב. GitHub זה אתר שמארח את המאגרי גיט הפרטיים ומאפשר לשתף עם אחרים.

•	Fetch רק מוריד את המידע החדש מהשרת למחשב כדי שנראה מה השתנה בלי שנוגע בקבצים. Pull עושה אוטומטית fetch ואחריו merge כדי לדחוף את העדכונים ישירות לקוד.

•	Merge משלב ענפים, יוצר קומיט מיזוג ושומר על היסטוריה עם ענפים. Rebase מעביר את הקומיטים ושם אותם מחדש בקצה של הענף המעודכן בשביל היסטוריה לינארית.

•	זה המקום שמאפשר לבחור איזה קבצים יכנסו לקומיט במקום שגיט יקח אוטומטית הכל בבת אחת.

•	Reset מוחק את הקומיט מההיסטוריה. Revert יוצר קומיט חדש שמבטל את הפעולות של הקומיט הישן.

•	קורה כשהענף הראשי לא התקדם מאז שהתפצלו ממנו. במקום ליצור קומיט מיזוג חדש גיט פשוט שם את המצביע של הענף הראשי איפה שהקומיט האחרון.

•	זה קובץ טקסט שמגדיר לגיט איזה קבצים או תיקיות להתעלם מהם ולא להעלות אותו. אם מוסיפים קובץ שבמעקב לגיטאגינור גיט ימשיך לעקוב אחריו ולכן כדי להסיר ממעקב משתמשים ב git rm –cached.

•	תג רגיל הוא מצביע לקומיט מסויים לשימוש אישי וזמני. תג מוער זה אובייקט ממש שנשמר בגיט וכולל מטא דאטא, מיועד לשחרור גרסאות רשמיות.

b.	רמה בינונית:

•	Soft מוחק את הקומיט ומשאיר את השינויים ב staging ובתיקיית העבודה. Mixed  מוחק את הקומיט ומוציא את הקבצים מה staging אבל השינויים נשארים בתקיית העבודה. Hard מוחק את הקומיט ואת כל השינויים מה staging ומתקיית העבודה.

•	זה מצב שבו ה HEAD מצביע ישירות ל hash של קומיט במקום על ענף. קומיטים באיזור הזה יהיו באוויר ויעלמו כשעוזבים. נכנסים בטעות כשעושים checkout ישירות ל hash ישן או כשנכנסים לעבוד בתוך תיקיית submodule בלי לפתוח בה ענף.

•	כשרוצים להעתיק קומיט ספציפי מענף אחד לאחר בלי למזג את כל שאר הענף. לדוג כשעובדים על ענף פיתוח ארוך ותוך כדי יצא לתקן שם באג. נצרך עכשיו את תיקון הבאג אז עושים cherry-pick רק לקומיט של התיקון ישירות ל main בלי לשחרר את שאר הפיתוח שעוד לא הסתיים.

•	פותחים את הקובץ שהתנגש בעורך קוד כלשהו. מוצאים את הסימני התנגשות של גיט. מחליטים איזה קוד לשמור (או שילוב) ומוחקים את הסימנים. מריצים git add על הקובץ כדי להראות לגיט שהבעיה נפתרה ומעלים קומיט.

•	בקומיט רגיל הם אותו הדבר, אבל בקומיט מיזוג- שיש לו 2 הורים, הם שונים: \~ הולך אחורה לדור הקודם על אותו ציר זמן. ^ מאפשר לבחור לאיפה מבין ההורים.

•	הפקודה דורסת את השרת. אם מישהו העלה קוד לפניי אני אמחק לו את העבודה בלי אזהרה. לכן משתמשים ב git push --force-with-lease שבודק קודם בשרת שאף אחד לא דחף שינויים ורק אם הכל נקי הוא דורס.

•	זה כמו מגירה ששומרת שינויים שלא עברו קומיט, ומנקה את אזור העבודה. זה טוב כשאני באמצע עבודה על פיצ'ר ופתאום צריך לקפוץ לענף אחר כדי לעשות משהו דחוף אבל לא רוצים לעשות קומיט לקוד לא גמור.

•	Merge commit מעביר את כל הקומיטים ויוצר קומיט מיזוג.

Rebase מעביר את כל הקומיטים בשורה אחת ליניארית בלי קומיט מיזוג.

Squash מועך את כל הקומיטים של הענף לקומיט אחד ב main.

c.	רמה מתקדמת:

•	משתמשים בפקודה git reflog כדי למצוא את ה hash של הקומיט שנאבד, ומשחזרים אותו בעזרת פתיחת ענף חדש שמצביע עליו- git branch salvage <hash>.

•	זה כלי לאיתור באגים שעובד על חיפוש בינארי. במקום לבדוק קומיט אחד אחד הוא חותך ת כמות הקומיטים לבדיקה בחצי בכל שלב.

•	הכלל הוא שאסור אף פעם לעשות rebase לענף פומבי שמישהו אחר כבר משך אליו. הסיבה היא כי rebase משכתב את ההיסטוריה ומשנה את ה hash. חברי צוות שכבר עובדים על הגרסה הישנה יתקעו עם קונפליקטים, קוד שבור וקומיטים כפולים כשינסו להסתנכרן.

•	כשמגלים באג ששייך לקומיט ישן, במקום ליצור קומיט רגיל, יוצרים קומיט שנשען על הישן- git commit --fixup <hash>. כשמריצים אחר כך git rebase -i --autosquash גיט ממקם את אותו הקומיט אוטומטית בדיוק מתחת לקומיט המקורי שהוא אמור לתקן ומחבר אותם יחד לקומיט אחד.

•	כמו יומן שמתעד את ההיסטוריה של HEAD במחשב. הוא שומר את ה hash של קומיטים שנמחקו וזה מאפשר לחזור אחורה ולתקן טעויות.

•	ענף main נעול ותמיד מכיל קוד עובד שאפשר להעלות לייצור. עבור כל משימה פותחים ענף קטן נוסף. בסיום הפיתוח מגישים PR ל Main שדורש ביקורת קוד ממתכנת אחר נוסף. ממזגים בעזרת squash כדי לשמור על הסטוריית main נקייה. השיקולים הם שזה מונע היווצרות של ענפים ארוכים שקשה למזג ועוזר לקוד להיות איכותי יותר בגלל ה PR, ושומר על הענף הראשי תמיד יציב.
